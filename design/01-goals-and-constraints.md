# 目标与约束

---

## 1. 项目目标

- **定义 AI 外呼系统的整体架构**，为后续设计、开发与迭代提供统一参照。
- 确立**通用内核 + 领域适配**的架构思路，系统不绑定单一行业，核心通话引擎与业务逻辑解耦。
- 明确**当前第一落地场景为房产线索初筛**，以此驱动 MVP 的功能范围和优先级。
- 承认当前的现实条件：**单人开发、成本敏感、快速验证**，架构选型和工程决策均以此为出发点。
- **预留开源到商业的切换路径**，核心引擎开源，商业能力（多租户、计费、高级分析等）以插件或独立模块形式叠加。

### 不包含

- 未经验证的收益承诺或 ROI 结论。
- 大厂式平台设计（如多区域部署、万级并发、企业级权限体系等）。
- 与当前阶段无关的远期规划细节。

---

## 2. 当前阶段目标

### 2.1 最小可运行闭环 — 核心能力列表

MVP 阶段需要跑通的完整闭环包括以下能力：

| # | 能力 | 说明 |
|---|------|------|
| 1 | 外呼发起 | 从任务队列取号、通过 SIP 中继发起呼叫 |
| 2 | 实时语音交互 | ASR 实时转写 → LLM 生成回复 → TTS 合成语音，全链路流式 |
| 3 | 话术引导 | 基于预设话术模板（房产场景），引导对话完成信息收集 |
| 4 | 意向判定 | 通话过程中或结束后，对线索意向进行分级（高/中/低/无效） |
| 5 | 通话记录 | 录音存储、ASR 转写文本留存 |
| 6 | 结果通知 | 高意向线索通过飞书/微信等渠道实时通知业务人员 |
| 7 | 任务管理 | 批量导入号码、设定外呼时段、查看任务进度 |
| 8 | 基础看板 | 接通率、平均通话时长、意向分布等关键指标可视化 |

### 2.2 需要验证的 5 个核心问题

1. **延迟体验**：ASR → LLM → TTS 全链路延迟能否控制在用户可接受范围内（目标 < 1.5 秒）？
2. **对话质量**：LLM 在房产初筛场景下，能否稳定完成多轮引导并准确提取关键信息？
3. **成本模型**：单通电话的综合成本（通信 + ASR + LLM + TTS）是否在商业可行区间？
4. **通用性验证**：核心引擎在切换到非房产场景（如教育、保险）时，需要改动多少？
5. **稳定性底线**：在 AI 组件异常时，系统能否优雅降级而不是让通话卡死？

### 2.3 房产是首个业务模板，不是系统边界

房产线索初筛被选为第一落地场景，原因是：

- 场景结构化程度高（预算、区域、户型、时间），适合 AI 引导式对话。
- 行业对外呼接受度相对较高，合规风险可控。
- 意向判定标准明确，验证闭环短。

但架构设计时，房产相关的逻辑（话术模板、意向判定规则、字段提取 schema）应作为**业务模板层**存在，与核心通话引擎严格分离。系统边界是「AI 外呼引擎」，而非「房产外呼工具」。

## 3. 当前约束与前提

### 3.1 开发前提

- **单人全栈开发**，无专职前端、运维或测试人员。
- 技术选型基于 **Go 生态**，利用其在并发、部署、性能方面的优势，标准库优先。
- 优先使用 Go 官方标准库及社区维护良好的第三方库，避免引入过多外部依赖。
- 优先使用现有开源组件拼装，避免从零造轮子。
- 配置格式统一使用 **TOML**（clarion.toml），不混用 INI/YAML 等格式。
- 代码组织以模块清晰、可独立替换为原则，不追求微服务粒度。
- CI/CD 从简，但必须有基础的自动化测试和部署流程。

### 3.2 通信前提

- 使用 **SIP 中继**接入运营商线路，通过 FreeSWITCH 或类似软交换处理信令与媒体。
- 初期支持**单并发或低并发**（1-5 路同时通话），不做高并发优化。
- 通话录音本地存储，后续按需对接 OSS。
- 号码资源和线路合规由业务侧负责，系统提供必要的频控和黑名单机制。
- 支持基础的呼叫控制：发起、挂断、转接（人工接管）。

### 3.3 AI 能力前提

#### ASR（语音识别）

- **MVP 阶段优先使用云端 API**（阿里云语音识别），降低开发复杂度和延迟风险。
- 云端方案成熟度高，开箱即用，支持流式识别，适合快速验证。
- **保留本地部署能力**（FunASR），用于后续成本优化阶段切换。
- 本地部署在延迟优化和长期成本控制上有优势，但初期引入会增加运维和调优负担。

#### TTS（语音合成）

- **MVP 阶段优先使用云端 API**（阿里云语音合成），保证合成质量和开发效率。
- 云端 TTS 在音色自然度、情感表达上通常优于自部署方案。
- **保留本地部署能力**（CosyVoice），用于后续成本优化或定制音色需求。

#### LLM（大语言模型）

- 使用 **DeepSeek API**，流式调用。
- DeepSeek 在中文场景下性价比高，响应质量满足初筛对话需求。
- 通过 prompt 工程实现话术引导和信息提取，初期不做微调。

#### 接口抽象要求

- **所有 AI 模块（ASR / TTS / LLM）必须做接口抽象**。
- 定义统一的流式调用接口，上层业务代码不直接依赖具体供应商 SDK。
- 支持通过配置切换供应商，无需改动业务逻辑。
- 流式接口为一等公民，批处理接口作为补充（用于后处理场景）。

### 3.4 抽象化前提

系统需要在以下层面保持抽象，避免与特定实现强耦合：

| 抽象层 | 说明 | 当前实现 | 可替换为 |
|--------|------|----------|----------|
| ASR 接口 | 统一流式语音识别接口 | 阿里云 ASR API | FunASR 本地 / Google STT / Azure |
| TTS 接口 | 统一流式语音合成接口 | 阿里云 TTS API | CosyVoice 本地 / Azure TTS |
| LLM 接口 | 统一流式对话接口 | DeepSeek API | OpenAI / Claude / 本地模型 |
| 通信层 | SIP 信令与媒体处理 | FreeSWITCH（ESL 协议） | Asterisk / Otel 等 |
| 业务模板 | 话术、字段定义、意向规则 | 房产初筛模板 | 教育/保险/通用模板 |
| 通知渠道 | 结果推送通道 | 飞书 Webhook | 微信 / 钉钉 / 短信 |
| 存储层 | 录音与数据持久化 | 本地文件 + SQLite/PostgreSQL | OSS + 云数据库 |

抽象的目标不是过度设计，而是确保在验证完 MVP 后，能以最小代价切换供应商或部署方式。

## 4. 设计原则

以下十条原则指导系统各模块的设计与实现决策。

### 4.1 实时优先

> 系统设计以单通电话的实时体验为第一优先级。

- 所有架构决策在「实时通话体验」和「系统复杂度」之间发生冲突时，优先保障实时体验。
- 延迟预算明确分配：ASR 识别延迟、LLM 首 token 延迟、TTS 首包延迟，各环节有独立指标。
- 非实时需求（报表、分析、通知）不得占用通话链路资源。

### 4.2 流式贯穿

> ASR / LLM / TTS 全链路流式处理，批处理仅用于后处理。

- ASR 输出实时文本流 → LLM 接收文本流并流式生成回复 → TTS 接收文本流并流式合成音频。
- 各环节之间通过流式接口串联，避免「等一个环节全部完成再启动下一环节」的瀑布式处理。
- 批处理模式仅用于通话结束后的摘要生成、数据落库、报表计算等后处理任务。

### 4.3 热冷分离

> 通话中只做最小闭环，摘要 / 落库 / 通知异步后处理。

- **热路径**（通话进行中）：只包含 ASR → LLM → TTS 核心对话循环，以及必要的状态维护。
- **冷路径**（通话结束后）：摘要生成、意向判定精调、CRM 落库、通知推送、录音归档等操作异步执行。
- 热路径中的任何操作都必须在毫秒级完成，不允许阻塞式 I/O 或重计算。
- 冷路径任务失败不影响通话体验，可重试。

### 4.4 故障友好

> 任何 AI 组件超时或失败不应导致通话卡死，必须有降级路径。

- **ASR 超时**：降级为静音检测，在超时后播放兜底话术（如「不好意思，我没有听清，您可以再说一次吗？」）。
- **LLM 超时**：使用预设的兜底回复，或根据当前对话阶段播放过渡语（如「好的，我记下了」）。
- **TTS 超时**：降级为预合成的兜底音频片段。
- **全链路异常**：优雅挂断并记录异常现场，通话标记为「异常中断」，进入重试队列。
- 每个 AI 调用必须设置超时上限，不允许无限等待。

### 4.5 幂等安全

> 防重复外呼、通知幂等、操作可重试。

- 同一号码在同一任务中不会被重复呼叫，除非显式标记为「需要重试」。
- 通知推送带唯一标识，接收端可去重，避免同一通话结果重复通知。
- 所有写操作（状态变更、数据落库）设计为幂等，重试不会产生副作用。
- 任务调度器具备防重机制，即使进程重启也不会导致重复执行。

### 4.6 接口抽象

> 所有 AI 能力通过统一流式接口暴露，供应商可替换。

- 定义 ASR / TTS / LLM 的统一接口协议，上层代码面向接口编程。
- 每个 AI 能力的具体实现作为适配器（Adapter）存在，通过配置注入。
- 接口设计以流式为主，包含：初始化、推送数据、接收结果流、关闭 四个基本操作。
- 新增供应商只需实现适配器，不需要修改通话引擎或业务逻辑。

### 4.7 可观测

> 通话全程事件可记录、可回放、可度量延迟。

- 通话过程中的关键事件（呼叫发起、接通、ASR 结果、LLM 响应、TTS 播放、挂断等）以结构化日志记录。
- 每个事件带精确时间戳，支持事后回放完整对话流程和延迟分析。
- 关键延迟指标可度量：
  - ASR 识别延迟（用户说完到文本输出）
  - LLM 首 token 延迟（文本输入到首个回复 token）
  - TTS 首包延迟（文本输入到首个音频包）
  - 端到端延迟（用户说完到系统开始播放回复）
- 异常事件（超时、降级、错误）单独标记，支持按异常类型筛选和告警。
- 初期不要求完整的分布式追踪体系，但事件日志格式需预留 trace_id 字段，便于后续接入。

### 4.8 测试先行

> 每个模块必须先有测试，再有实现。

- 所有模块在实现前必须编写对应的测试用例，测试驱动开发。
- **Table-driven tests** 作为标准测试模式，覆盖正常路径、边界条件和异常场景。
- 热路径代码（ASR/LLM/TTS 流式处理、状态机转换等）必须编写 **benchmark 测试**，防止性能回退。
- 测试覆盖率目标 **> 80%**，核心引擎模块要求更高。

### 4.9 性能可度量

> 所有热路径操作必须有基准测试，性能可量化、可追踪。

- 热路径操作（音频处理、状态机转换、流式数据传递等）必须有对应的 **benchmark 测试**，作为性能基线。
- 支持通过 **pprof / trace** 进行运行时性能剖析，便于定位瓶颈。
- 关键延迟指标通过**结构化日志**暴露，支持外部采集和告警。

### 4.10 配置统一

> 单一 TOML 配置文件，环境变量仅用于敏感信息覆盖。

- 系统使用单一配置文件 **clarion.toml**，所有模块的配置集中管理。
- **环境变量**仅用于覆盖敏感信息（数据库密码、API Key 等），不作为主要配置方式。
- 不混用 INI / YAML / TOML 等多种配置格式，保持配置来源单一可追溯。
